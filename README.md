# Log Analyzer CLI

یک ابزار خط فرمان (CLI) ساده، سریع و بهینه برای تحلیل فایل‌های `access log` سرویس‌های وب. این ابزار برای مدیریت فایل‌های حجیم طراحی شده و با استفاده از پردازش خط‌به‌خط، مصرف حافظه (RAM) را به حداقل می‌رساند.

## ویژگی‌ها
* **پارس کردن سریع:** استفاده از Regular Expressions برای تجزیه فرمت Combined Log.
* **بهینه‌سازی حافظه:** پردازش فایل‌ها به صورت Streaming (خط‌به‌خط) که امکان کار با فایل‌های چند گیگابایتی را فراهم می‌کند.
* **مدیریت خطا:** شناسایی و پرش از خطوط خراب (Malformed) بدون کرش کردن برنامه.
* **پشتیبانی از فرمت فشرده:** قابلیت خواندن مستقیم فایل‌های `.gz`.
* **گزارش‌دهی هوشمند:**
    * آمارهای پایه (درخواست‌های کل، آی‌پی‌های یکتا، نرخ خطا).
    * لیست ۱۰ اندپوینت پربازدید (قابل تنظیم).
    * توزیع زمانی ترافیک (هیستوگرام متنی).
    * شناسایی فعالیت‌های مشکوک (تلاش برای نفوذ).
    * تشخیص جهش خطاهای ۵xx (Error Spike Detection).
* **خروجی منعطف:** خروجی متنی خوانا برای انسان و فرمت JSON برای سیستم‌های مانیتورینگ.

## نحوه اجرا

برای استفاده از ابزار، کافیست دستور زیر را در ترمینال وارد کنید:

```bash
python log_analyzer.py path/to/access.log
```

## آپشن‌های اضافه:
* تغییر تعداد اندپوینت‌های برتر:

```bash
python log_analyzer.py access.log --top 20
```

* دریافت خروجی JSON (مناسب برای Pipeline):

```bash
python log_analyzer.py access.log --json
```

## تصمیمات مهندسی
* پردازش: به دلیل حجم بالای فایل‌های لاگ، از `generator`ها استفاده کردم تا فایل به صورت کامل در حافظه بارگذاری نشود.
* ساختار: از `argparse` برای رابط کاربری استاندارد و از `collections.Counter` برای بهینه‌سازی عملیات شمارش استفاده شده است.
* پایداری: تمامی عملیات فایل‌خوانی در بلاک `try-except` قرار گرفته تا از متوقف شدن برنامه در مواجهه با فایل‌های خراب جلوگیری شود.

## چالش‌ها و حل مسئله
* مشکل: مدیریت خطوط "کثیف" و ناقص در لاگ‌های واقعی.
راه‌حل: در حین پارس کردن، به جای `match.group()` مستقیم، از منطق شرطی `(if not match: continue)` استفاده کردم تا خطوطی که با فرمت اصلی همخوانی ندارند شمارش شده و نادیده گرفته شوند، بدون اینکه روند تحلیل متوقف شود.
